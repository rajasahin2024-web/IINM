# MIGRATION SAFETY — NON-NEGOTIABLE

## ❌ NEVER ALLOW
- `drop_all()`, `Base.metadata.drop_all()`, or any table drop
- `op.drop_table()` in any migration
- `op.drop_column()` without explicit user approval AND backup confirmation
- [downgrade()](cci:1://file:///www/wwwroot/iinm/backend/migrations/versions/20260508_1421_8fbbb2dea4a9_initial_baseline_all_tables.py:45:0-49:8) that drops columns/tables on production
- Running `alembic downgrade` on production database
- Manual `DELETE FROM` or `TRUNCATE` in migration scripts
- Any migration that modifies existing data without backup

## ✅ ALWAYS DO
1. **Backup before any migration**: `cp data.db data.db.backup.$(date +%Y%m%d_%H%M%S)` (SQLite) or `pg_dump` (PostgreSQL)
2. **Review autogenerated migrations** before running — never auto-approve `alembic upgrade`
3. **Only safe operations allowed by default**:
   - `op.add_column()` — adding new nullable columns
   - `op.create_table()` — creating entirely new tables
   - `op.create_index()` — adding indexes
   - `op.alter_column()` with `server_default` — safe defaults
4. **Test on staging/copy first** before touching production
5. **Use `nullable=True` and `server_default`** for new columns, never require immediate data

## 🚨 RED FLAGS — STOP IMMEDIATELY
If a migration contains ANY of these, STOP and ask user:
- `DROP`
- `drop_all`
- `delete from`
- `truncate`
- `cascade`
- `reset`
- `recreate`

## PORT RULE
Frontend admin pages must use `BASE_URL` from config ([/lib/config.ts](cci:7://file:///www/wwwroot/iinm/frontend/src/lib/config.ts:0:0-0:0)), NEVER hardcoded ports like `:8000`.
